# TSFinancialSystem 技术路线图

更新时间：2026-06-29

本文档用于记录 TSFinancialSystem 的后续工程路线、优先级、验收标准和文档分工。它面向项目维护者和开发者；面向平台用户的说明应放在 `README.zh-CN.md` / `README.md`，架构细节放在 `ARCHITECTURE.md`。

## 1. 当前定位

TSFinancialSystem 当前已经从“个人资产记账工具”演进为一个自部署的个人投资 cockpit，核心能力包括：

- 多平台、多币种、多资产类别的资产管理。
- 交易流水驱动持仓、成本和已实现盈亏。
- 行情、汇率、净值和资产配置分析。
- CSV 导入、预览校验、去重和对账。
- 投资决策日志、AI 投研报告和行动项跟踪。
- BYOK、多用户隔离、隐私打码、Docker / Fly.io 部署。

后续迭代不应继续横向堆功能，而应优先提升四件事：

1. 数据可信度：计算准确、来源清楚、状态可解释。
2. 冷启动效率：用户能更容易导入历史交易并完成对账。
3. 投研闭环：AI 输出能变成可跟踪、可复盘的投资动作。
4. 工程可维护性：模型、服务、迁移、测试和部署更稳定。

## 3. Phase 0：稳定性与可信度补强

目标：把系统从“功能完整”提升为“数据可信、逻辑稳定、可长期使用”。

| 优先级 | 任务 | 产品价值 | 工程价值 | 状态 |
| --- | --- | --- | --- | --- |
| P0 | 修复 Portfolio Review 多币种混算风险 | 避免 AI 对组合权重做出错误判断 | 统一估值口径 | 待做 |
| P0 | 普通账户禁止超卖 | 避免持仓数量和收益计算出现明显错误 | 明确交易规则 | 待做 |
| P0 | 金额、价格、数量字段 Decimal 化 | 提高金融计算可信度 | 减少浮点误差 | 待做 |
| P0 | 修复 BYOK 后台任务 retrieve / cancel | 保证 AI 任务可恢复、可取消 | 完善 provider 状态管理 | 待做 |
| P0 | 数据状态中心 | 让用户理解行情、汇率、手填资产是否可信 | 汇总数据新鲜度与错误状态 | 部分完成 |
| P0 | GitHub Actions CI | 防止回归 | 建立质量门禁 | 待做 |

验收标准：

- 混合 CNY / USD / HKD 持仓时，组合权重必须先折算到统一展示币种。
- AI prompt 中必须标注金额币种、汇率时间和数据时间点。
- 普通账户卖出数量不得超过当前可用数量；CSV preview 阶段必须标红超卖风险。
- 后端测试覆盖买入、部分卖出、全部清仓、超卖拒绝、历史交易修改导致的超卖。
- CI 至少包含后端 `pytest`、前端依赖安装、前端构建和 Docker build smoke test。

## 4. Phase 1：导入与对账能力

目标：降低用户冷启动成本，让系统从“手动维护工具”升级为“可对账资产账本”。

| 优先级 | 任务 | 说明 | 状态 |
| --- | --- | --- | --- |
| P1 | 富途 CSV 导入 | 覆盖常见中文券商导出格式 | 部分完成 |
| P1 | IBKR Activity Statement 导入 | 覆盖海外券商核心场景 | 部分完成 |
| P1 | 通用字段映射器 | 支持用户手动确认字段映射 | 部分完成 |
| P1 | 导入 preview 校验 | 展示 valid / warning / error / duplicate | 部分完成 |
| P1 | 导入后对账页面 | 对比券商持仓和系统回放持仓 | 部分完成 |
| P1 | 异常交易修正入口 | 支持 ignored / corrected / manual adjustment | 待做 |
| P1 | 现金账本完善 | 入金、出金、买卖联动现金余额 | 部分完成 |

验收标准：

- 导入前不直接写入交易表，必须先 preview。
- preview 能识别重复交易、字段缺失、币种异常、价格缺失、超卖风险。
- commit 后必须触发交易回放并更新 derived 持仓。
- 对账页能展示数量差异、成本差异、现金差异和需要人工处理的项目。
- 导入过程可重复执行，不产生重复交易。

## 5. Phase 2：投研闭环增强

目标：让 AI 报告不只是文本输出，而是进入“行动项 - 观察 - 复盘 - 决策”的闭环。

| 优先级 | 任务 | 说明 | 状态 |
| --- | --- | --- | --- |
| P2 | AI 报告行动项结构化 | 将 action items 转为可跟踪对象 | 已完成基础版 |
| P2 | 持仓研究摘要 | 在持仓详情中聚合 thesis / risk / review / report | 已完成基础版 |
| P2 | 决策日志状态流转 | active / resolved / invalidated / archived | 已完成基础版 |
| P2 | 复盘提醒 | 对长期未处理行动项生成提醒 | 待做 |
| P2 | 报告数据快照 | 固化报告生成时的行情、持仓和汇率上下文 | 待做 |
| P2 | 投研质量检查清单 | 在报告中强制输出假设、反证、风险和待验证问题 | 部分完成 |

验收标准：

- 每份 AI 报告能追溯生成时的数据时间点、币种口径和关键输入。
- 行动项能关联标的、持仓、报告和状态。
- 用户能从持仓页看到相关 thesis、risk、review、action item 和历史报告。
- 报告必须明确“不构成投资建议”，并区分事实、假设和待验证问题。

## 7. Phase 4：部署与运维质量

目标：让项目更适合长期自托管、多人使用和公开展示。

| 优先级 | 任务 | 说明 | 状态 |
| --- | --- | --- | --- |
| P2 | 生产环境健康检查 | 增加 DB、行情、汇率、邮件、AI provider 状态 | 待做 |
| P2 | 备份恢复演练文档 | 明确 JSON 备份、数据库备份、Fly volume 备份流程 | 待做 |
| P2 | 日志结构化 | 让导入、刷新、AI 任务和邮件问题更容易定位 | 待做 |
| P3 | Sentry / OpenTelemetry 可选接入 | 自托管用户可选择开启错误追踪 | 待做 |

验收标准：

- 生产环境启动时能发现关键 secret 缺失，并给出明确错误。
- 用户能在文档中找到备份、恢复、迁移 PostgreSQL 的路径。
- 自动刷新失败、AI 任务失败、邮件发送失败都有可读日志和前端状态反馈。

## 8. 优先级原则

后续开发按以下原则排序：

1. 先修正会影响资产准确性的逻辑，再新增页面。
2. 先让导入、回放、对账稳定，再做 Broker API 直连。
3. 先沉淀 AI 报告的可追踪上下文，再增加更多 AI 模板。
4. 先建立 CI、测试和迁移，再做大规模模型拆分。
5. README 面向用户，架构文档面向开发者，路线图面向维护者，避免重复堆叠。
