# future_update_6_28

## 1. 文件目的

本文件用于记录 **TSFinancialSystem** 后续迭代路线，基于 2026-06-28 对系统的产品经理视角与研发视角审查结果整理。

当前系统已经不是简单的资产记账工具，而是一个具备以下能力的 **个人资产管理 + 投研记录 + AI 辅助分析系统**：

1. **多平台资产管理**
2. **多币种资产汇总**
3. **交易驱动持仓**
4. **行情与汇率刷新**
5. **Dashboard 资产总览**
6. **AI 投研报告**
7. **投资决策日志**
8. **提醒系统**
9. **Docker / Fly.io 部署**
10. **SQLite / PostgreSQL 可选支持**

后续迭代不应继续横向堆功能，而应围绕两个核心资产强化：

1. **交易驱动持仓引擎**
2. **AI 投研到行动项的闭环**

## 2. 当前系统定位

### 2.1 产品定位

**TSFinancialSystem** 的推荐定位是：

> 一个面向个人投资者的 **多币种资产聚合、交易账本、投研复盘与 AI 决策辅助 cockpit**。

它不应被包装成普通记账系统，而应突出以下差异点：

1. **交易驱动持仓**
   - 持仓数量、成本、已实现盈亏由交易流水自动回放生成。
   - 这比手动维护持仓更接近真实资产管理逻辑。

2. **多币种资产总览**
   - 支持 CNY / USD / HKD 等币种。
   - 适合拥有海外券商、美股、基金、债券、现金资产的用户。

3. **AI 投研闭环**
   - AI 报告不是孤立文本。
   - 报告中的行动项可以沉淀为决策日志。
   - 持仓页面可以反查相关笔记、报告和行动项。

4. **自托管与隐私**
   - 用户资产数据不需要托管给第三方 SaaS。
   - 支持 BYOK，即用户配置自己的 AI API Key。

## 3. 当前主要优势

### 3.1 产品优势

1. **用户问题真实**
   - 个人资产常分散在多个券商、银行、钱包和基金账户。
   - 系统已经围绕“多平台、多币种、多资产类别”构建。

   - 添加或导入交易。

   - 刷新行情和汇率。
   - 查看资产总览。
   - 生成 AI 投研报告。

   - 后续可以发展成投资复盘系统，而不仅是聊天式问答。

2. **交易驱动持仓设计正确**

## 4. 当前主要问题

### 4.1 产品问题

#### 4.1.1 功能已经偏多

当前系统功能已经覆盖资产、交易、投研、笔记、提醒、自动刷新、AI Key、邮箱验证、备份恢复等模块。

短期内不建议继续新增大功能。下一阶段重点应从 **功能数量** 转向 **数据可信度、导入体验、计算准确性、复盘闭环**。

#### 4.1.2 冷启动门槛仍然偏高

资产管理产品的最大阻碍不是 Dashboard，也不是 AI 报告，而是：

1. 用户如何把历史交易导入系统。
2. 导入后系统算出的持仓是否和券商一致。
3. 后续是否需要大量手动维护。
4. 出错时能否快速定位和修正。

因此，后续必须优先做：

1. **富途 CSV 导入**
2. **IBKR CSV / Activity Statement 导入**
3. **导入后对账**
4. **异常交易修正**

#### 4.1.3 数据可信度需要产品化

系统已有行情刷新和数据新鲜度字段，但还需要一个更明确的 **数据状态中心**。

需要向用户展示：

| 数据类型 | 需要展示的信息 | 目的 |
|---|---|---|
| **股票行情** | 数据源、更新时间、是否过期 | 避免用户误以为数据实时 |
| **基金净值** | 净值日期、更新时间 | 避免场外基金被误解为实时价格 |
| **汇率** | 来源、更新时间、缓存状态 | 让多币种折算可解释 |
| **手填资产** | 用户最后更新时间 | 避免长期未更新 |
| **AI 报告** | 数据时点、假设、待验证问题 | 避免 AI 结论被误当成确定事实 |

---

### 4.2 研发问题

#### 4.2.1 `models.py` 已经过重

当前 `backend/models.py` 同时承担：

1. 数据库表模型
2. API Schema
3. 枚举
4. 用户安全问题
5. 投研报告模型
6. AI Key 模型
7. Alert 模型
8. 市值、成本、盈亏计算函数

这会增加后续维护成本。建议后续拆分。

推荐拆分方向：

| 文件 | 职责 |
|---|---|
| `models/user.py` | User、AuthToken、安全问题 |
| `models/portfolio.py` | Platform、Holding、Transaction、Snapshot、FxRate |
| `models/research.py` | ResearchReport、Note |
| `models/alerts.py` | AlertRule、AlertEvent |
| `schemas/` | API 输入输出 Schema |
| `services/valuation.py` | market_value、day_change、cost_basis、profit |
| `services/position.py` | 交易回放和持仓重算 |

#### 4.2.2 数据库迁移方式不适合长期生产

当前采用 `SQLModel.metadata.create_all` 加轻量补列的方式处理数据库结构变化。

短期可用，但长期不适合生产。建议引入 **Alembic**，实现版本化 migration。

#### 4.2.3 金融字段不应长期使用 `float`

涉及金额、价格、费用、数量的字段应逐步迁移到 **Decimal / Numeric**。

优先级较高的字段：

1. `quantity`
2. `manual_value`
3. `cost_price`
4. `current_price`
5. `prev_close`
6. `fee`
7. `amount`
8. `realized_pnl`
9. `realized_income`

#### 4.2.4 超卖逻辑需要调整

当前系统允许卖出数量超过持仓后出现负数量。

建议短期策略：

1. 普通账户默认禁止超卖。
2. 如果未来支持融资融券 / 做空，再增加账户类型。
3. 超卖时返回明确错误提示。
4. CSV 导入时如出现超卖，应在 preview 阶段标红。

#### 4.2.5 AI BYOK 后台任务需要修复

当前 AI 报告生成支持用户自带 Key，但后台刷新 / 取消任务时需要确保使用当初创建任务的用户 Key，而不是依赖系统级环境变量。

建议：

1. `ResearchReport` 增加 `user_ai_key_id` 或 key reference。
2. `refresh_run` 时重新读取并解密对应 Key。
3. `cancel_report` 时也使用对应 Key。
4. provider / model / base_url / key reference 一起保存，保证任务可恢复。

#### 4.2.6 Portfolio AI 上下文存在币种混算风险

Portfolio Review 中不能直接把不同币种的 `market_value` 相加。

应统一使用 valuation service：

1. 持仓原币种市值。
2. CNY 折算市值。
3. USD 折算市值。
4. 组合权重统一基于展示币种或 CNY 基准计算。
5. AI prompt 中明确标注每个金额的币种和汇率时间。

## 7. Phase 1：导入与对账能力

### 7.1 目标

降低用户冷启动成本，使系统从“手动维护工具”升级为“可对账资产账本”。

### 7.2 P1 任务清单

| 优先级 | 任务 | 说明 |
|---|---|---|
| **P1** | 富途 CSV 导入 | 先支持用户最可能使用的数据源 |
| **P1** | IBKR Activity Statement 导入 | 覆盖海外券商核心场景 |
| **P1** | 导入字段映射器 | 允许用户手动匹配字段 |
| **P1** | 导入后对账页面 | 对比券商持仓和系统持仓 |
| **P1** | 异常交易修正入口 | 支持 ignored / corrected |
| **P1** | 现金账本 | deposit / withdraw 影响现金余额 |

### 7.3 导入流程设计

推荐流程：

1. 用户选择券商类型。
2. 上传 CSV / Activity Statement。
3. 系统自动识别字段。
4. 用户确认字段映射。
5. 系统进行 preview 校验。
6. 显示错误行和风险提示。
7. 用户确认导入。
8. 系统生成交易流水。
9. 系统回放生成持仓。
10. 系统进入对账页面。
11. 用户确认或修正差异。

### 7.4 对账页面字段

| 字段 | 说明 |
|---|---|
| **平台** | 富途 / IBKR / 老虎等 |
| **代码** | 股票、ETF、基金代码 |
| **名称** | 标的名称 |
| **券商数量** | 从导入文件读取 |
| **系统数量** | 交易回放计算 |
| **数量差异** | 券商数量 - 系统数量 |
| **券商成本** | 如文件支持则读取 |
| **系统成本** | 系统移动加权成本 |
| **成本差异** | 两者差额 |
| **状态** | matched / warning / error |
| **操作** | 查看交易、修正、忽略 |

## 8. Phase 2：AI 投研闭环强化

### 8.1 目标

让 AI 模块从“生成报告”升级为“投资决策复盘系统”。

### 8.2 产品闭环

目标闭环：

> **持仓数据 → AI 分析 → 行动项 → 提醒规则 → 跟踪记录 → 复盘报告**

### 8.3 P2 任务清单

| 模块 | 当前状态 | 下一步 |
|---|---|---|
| **Portfolio Review** | 已有基础能力 | 修复币种折算，增加集中度、风险暴露、收益来源分析 |
| **Thesis Tracker** | 已有 Note / action item | 增加 review_date、priority、evidence、status history |
| **Risk Monitor** | 已有提醒系统 | AI 报告风险项自动转 AlertRule |
| **Earnings Review** | 可生成报告 | 支持财报后假设复盘 |
| **Monthly Review** | 暂无固定闭环 | 自动生成月度资产变化与决策复盘 |
| **Holding Brief** | 已有持仓研究摘要 | 增加 thesis / risk / action / review 时间线 |

### 8.4 AI 报告结构建议

后续 AI 报告应统一输出以下结构：

1. **结论摘要**
2. **当前持仓状态**
3. **核心假设**
4. **主要风险**
5. **待验证问题**
6. **跟踪指标**
7. **行动项**
8. **下次复盘时间**
9. **数据时点与不确定性说明**

### 8.5 行动项结构化字段

建议把行动项从纯文本升级为结构化对象：

| 字段 | 说明 |
|---|---|
| **title** | 行动项标题 |
| **symbol** | 关联标的 |
| **source_report_id** | 来源 AI 报告 |
| **priority** | high / medium / low |
| **due_date** | 截止或复盘日期 |
| **status** | active / done / invalidated / archived |
| **evidence** | 支撑依据 |
| **created_at** | 创建时间 |
| **updated_at** | 更新时间 |
